## 5. 自動ループの仕組み（fail-closed 二重関所）

```
堤さんが回答（Webフォーム / メール返信 / LINE返信 のどれでも）
   → hs-hearing が正規化してKV保存（見積もり例は監査用・金額は生成に渡さない）
   → repository_dispatch: yakumo-hearing-completed
   → GitHub Action:
        generate.py  … GEO4/AEO3/LLMO2/WebMCP1 を /yakumo/ 配下に生成
        validate.py  … 関所A(fail-closed): JSON-LD/canonical/robots/禁止語MOAT(32.5,danger_threshold,WPC)/
                        禁止dash/金額なし/モール+HSバックリンク/裸相対リンク。1枚でも落ちたら commit しない
        commit+push  … main へ（Pages配信）+ sitemap追記
        indexnow     … 関所B: 各URL 200/MOAT無しを物理再チェックした分だけ IndexNow へ
   → 施主・検索・AI が新ページ経由でモールとHORIZON SHIELDに到達（認知度）
```
「検証通過なら公開まで全自動」＝関所A・Bを全通過したものだけ人手なしで公開。**だから絶対にGoogle/Bingにエラーを出さない。**

- 生成は `/yakumo/souba|faq|llmo|webmcp/` 配下のみ。本体の `/souba/` `/faq/` 等には一切書かない（重複ゼロ設計）。
- IndexNow は1日100件上限。1店=10ページなので余裕。

## 6. hs-hearing のエンドポイント一覧

公開:
- `GET /health` / `GET /.well-known/agent-card.json`
- `POST /mcp`（JSON-RPC: initialize/tools/list/tools/call）tools = `list_verified_stores` / `get_contractor_profile`（金額返さない）
- `GET/POST /h/<token>`（ヒアリングフォーム / 回答受信）
- `POST /line/webhook`（LINE。署名検証→登録コードで紐づけ→回答を自動構造化→生成）

管理（`X-Admin-Key: <HEARING_ADMIN_SECRET>`）:
- `POST /admin/provision`（店発行＋WebMCPオプション＋ヒアリングリンク）
- `POST /admin/send-hearing {token,to}`（ヒアリング案内メール・RESEND）
- `POST /admin/send-greeting {to,company?}`（初回あいさつメール・RESEND）
- `POST /admin/link-line {store_id,line_user_id}`（LINE手動紐づけ・予備）
- `GET /admin/export/<store_id>` / `GET /admin/hearing/<store_id>`（プロフィール/生回答）

メール返信の安全網: `email()` ハンドラが Email Routing 経由の返信を、件名 `ref:<token>` か送信元アドレスで店に照合→LLM構造化→必須項目チェック(fail-closed)→生成。
LINE返信: `line2store:<userId>` で照合→同じ ingestHearingAnswer→生成。
